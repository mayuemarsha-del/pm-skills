## 分项序号

**普通分项——反例：**
```markdown
修法：
- Skill：缩窄触发条件
- 知识：调召回阈值
```

**正例：**
```markdown
修法：
1. Skill：缩窄触发条件
2. 知识：调召回阈值
```

**表格单元格内——反例（粗体标签平铺，运营分不清边界）：**
```markdown
| 字段 | 口径 |
|------|------|
| 回答总数 | 机器人给出的回答数量。回答总数 = 资源回答数 + 智能回答数。**不计入**：所有缺省文案给出的回答。**任务流程**：一个任务流程只计 1 次 |
```

**正例（编号 + `<br>` 显式分隔）：**
```markdown
| 字段 | 口径 |
|------|------|
| 回答总数 | 机器人给出的回答数量。<br>1. **计算公式**：回答总数 = 资源回答数 + 智能回答数。<br>2. **不计入**：所有缺省文案给出的回答（如任务流程中各类缺省文案）。<br>3. **任务流程**：一个任务流程只计 1 次。 |
```

判断标准：单元格里出现 ≥ 2 个粗体小标题，或 ≥ 2 个语义独立句子，就换成编号格式。

## 重申提示语

默认不要写的典型反例：

1. 「客服助手和接待机器人均包含 X 和 Y 两种模式，指标适用性按模式判定，与机器人类型无关。」——重申矩阵已经表达过的事实
2. 「以上 5 项独立计数，可能相加超过 100%，运营理解时要注意。」——已经在分组标题、卡片提示里说过，正文又重述一遍
3. 「本节内容承接 X 节，详见 Y 节适用矩阵。」——除非真的有跨节歧义，否则不加
4. 任何「**注意**」「**说明**」「**提示**」开头的 callout，如果不是用户明确要求或关键风险点

例外见 SKILL.md 二、通用排版规则对应小节。

## 提示词不用表格

**反例（不要这样写）：**
```markdown
按 entry_type 分流：

| 入口形态 | 处理 |
|---|---|
| known_message_id | 调 fetch_log；日志缺失发 ask...；部分缺失标"数据缺失"跳过 |
| no_case | 发 ask...；用户回 ID → 转 known_message_id；用户回对话 → 直接归因 |
```

**正例：**
```markdown
按 entry_type 分流：

1. known_message_id
   1. 调 fetch_log 拿日志，进步骤二
   2. 日志缺失：发 ask...
   3. 部分缺失：标"数据缺失"跳过，其他维度照常

2. no_case
   1. 发 ask...
   2. 用户回 ID → 转 known_message_id 路径
   3. 用户回对话片段 → 直接归因
```

## 原型后不写 bullet

**反例：**
```markdown
（ASCII 主导航图）

- 左侧主导航：全局常驻，按「工作台 / 评测中心 / ...」分组
- 工作台是默认首页
- 底部「Close Menu」按钮可折叠为窄条
- 从其他模块点回「工作台」→ 恢复上次状态
```

**正例：** ASCII 图后直接进入下一个内容区块（下一张图 / 字段列表 / 操作逻辑）。规则和默认态全部归并到操作逻辑：

```markdown
### 操作逻辑

1. **进入工作台时**：
   1. 用户登录后默认进入工作台
2. **主导航交互**：
   1. 点击「Close Menu」时，主导航折叠为窄条
3. **从其他模块切回工作台时**：
   1. 恢复上次离开时的对话状态
```

## 操作逻辑

**基本格式：**
```markdown
#### 操作逻辑

1. **[操作名称]时**：
   1. [条件A]时，[系统行为]
   2. [条件B]时，[系统行为]

2. **点击「保存」时**，按字段顺序校验：
   1. **[字段名A]**：
      1. 为空时，提示"..."
      2. 超限时，提示"..."
   2. **[字段名B]**：[条件]时，提示"..."
   3. 以上校验全部通过时，[执行动作]
```

**校验类——反例（平铺多行）：**
```markdown
- 当同义词组名称为空时，阻止保存，提示"..."
- 当同义词组名称超过 50 字符时，阻止保存，提示"..."
- 当某行标准词为空时，...
```

**正例（按字段分组 + 编号，分组标题层也用序号）：**
```markdown
1. **同义词组名称**：
   1. 为空时，提示"..."
   2. 超过 50 字符时，提示"..."
2. **标准词**：
   1. 某行为空时，提示"..."
   2. 同组内重复时，提示"..."
```

## 多分支流程

**基本格式模板：**
```markdown
#### 4.X.Y [路径名 - 来源]

[一句话定位本节做什么]

**处理流程：**

1. **[步骤 A]**：[做什么]
2. **[步骤 B]**：[一句话]——
   1. [子项 1，含参数值如 N × 1.3]
   2. [子项 2，含边界场景：阻止生成 / 提示文案 / 兜底取值等]
   3. [子项 3，含数据结构如"单候选 = 字段甲 + 字段乙"]
       1. **[嵌套子项标题]：** [展开]
       2. **[嵌套子项标题]：** [展开]
3. **[步骤 C]**：
   1. [规则 1]
   2. [规则 2]
4. **添加至数据集**
```

**避免：**
1. 共享"默认参数表 + 共性边界表"+ 路径间互相引用（读者翻三处才能拼出完整逻辑）
2. "同 4.X.Y 第 N 步"压缩式跨节引用——只在 prompt 等长文本完全重复时用
3. 把多条规则塞成一行（"候选间相似度过滤 + 与数据集已有相似度过滤 + 字数 / 主题 / 乱码检测"）
4. 露内部实现术语（"种子""候选池""embedding 余弦"在边界提示文案里出现）

## 接待记录目录范例

层次原则「页面 → 区域 → 模块 → 分支」的示例：

```
3.2 列表页
    3.2.1 列表（含展示字段配置入口）
    3.2.2 导出
    3.2.3 数据订阅
3.3 会话详情
    3.3.1 左侧：历史会话列表
    3.3.2 中间：对话时间轴
        3.3.2.x 匹配链路
            1. 差异对比
            2. 自主决策模式
            3. 知识增强模式
            4. 三方机器人
    3.3.3 右侧：会话信息面板
```

读时一眼看出：列表页底下三块（列表 / 导出 / 订阅），会话详情按左 / 中 / 右分区，中间区域里的匹配链路再拆四个分支。不要把"匹配链路"和"详情"平级，也不要把"差异对比 / 自主决策"放到和"会话详情"同层。

## 流程类通用原则

### 锁定对象粒度，再写方案

反例：一上来写"新增一种知识库类型：表格知识"。
正例：先确认现状——知识库是容器层，"添加知识"是在知识库内新增知识内容；所以正确表达是"表格知识是知识库中的一种新增知识内容类型，而不是新的知识库类型"。
为什么：这直接影响入口位置、列表结构、详情页设计和后续交互。

### 先写页面，再写规则

反例：先写复杂检索逻辑、多条件组合、变量映射，再回头补页面。
正例：先写入口在哪 → 创建页 / 字段编辑页 / 详情页 / 编辑模式 → 最后才写变量映射、条件优先级、结果处理。
为什么：这样文档更像产品文档，不会写成算法方案。

### 字段约束收敛在字段表

反例：字段名长度写一处、必填规则写另一处、校验文案散落后文。
正例：字段表直接写"是否必填 / 格式限制 / 校验规则 / 说明"。
为什么：更好检索，也避免重复和冲突。

### 被纠正后整体重排

反例：用户补充一个关键现状后，只改一小段。
正例：如果这个现状影响对象层级、入口、页面框架，就整体重排相关章节。
为什么：结构错了，局部补丁只会越补越乱。

### 产品层 vs 实现层

产品文档定义：用户看到什么、入口在哪、状态怎么展示、页面怎么交互。
不过度暴露：Subagent 名称、内部执行编排细节、过细的算法判断树。
用产品语言表达。
